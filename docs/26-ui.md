# UI (`@rovy/ui`)

Status: planned, docs-only
Target: stateless widget package for Rovy
Goal: define a future UI package that fits Rovy's transformer model without adding React-style hooks or widget-local app state.

This document is a design spec, not shipped behavior.

`@rovy/ui` is planned as a distinct runtime package, parallel to `@rovy/networking`. It owns widget authoring, widget runtime plumbing, and any UI-specific registry/runtime helpers. The transformer owns compile-time discovery and callsite lowering.

## Core direction

The UI model is intentionally **stateless**.

It does **not** include:

- `useState`
- `useEffect`
- hook slots
- effect dependency tracking
- widget-local persistent app state
- a primary built-in surface like `RovyUi.Window(...)`

Instead, UI is authored from:

- widget call args
- injected resources
- commands / events
- explicit external stores owned elsewhere

If a widget changes over time, that change must come from the outside world. The widget itself is a pure/stateless builder over the current inputs.

## Package boundary

`@rovy/ui` is planned to ship:

- the `useWidget(...)`-style runtime plumbing
- the runtime widget registry / invocation surface
- any UI-specific base types needed for widget authoring
- built-in runtime support for widget tree mounting / reconciliation

`@rovy/core` remains ECS/runtime-focused. It still provides the general registration model and `rovy.loadPaths(...)`, but UI-specific authoring belongs in its own package.

`rovy-transformer` remains the compile-time piece. UI depends on it because raw TypeScript cannot directly express the intended plain-call widget syntax.

## `@widget`

Widgets are authored with two pieces in the same file:

- a normal caller function tagged with JSDoc metadata
- a builder class that contains the real widget implementation

The caller function exists so editor/typechecker behavior stays normal. The builder class exists so the runtime has an explicit implementation surface with a required `build(...)`.

```ts
/** @widget WindowBuilder */
export function Window(options: { title: string }): void {
	throw "transformer only";
}

@widget
class WindowBuilder {
	build(theme: Res<Theme>) {
		return (options: { title: string }) => {
			// widget runtime authoring goes here
		};
	}
}
```

The JSDoc tag explicitly names the builder. The transformer should resolve that builder in the **same file only**. No cross-file lookup and no naming-convention guessing.

Planned contract:

- the user-facing widget API is a normal function, not a class call
- caller metadata comes from JSDoc `@widget BuilderName`
- the named builder is a local `@widget` class in the same file
- each builder class is auto-registered through transformer-injected side effects
- the builder class must implement `build(...)`
- `build(...)` returns the renderer/function used by the widget runtime
- authored calls to the caller function stay plain function calls with normal args

## Auto-collection

Widget discovery follows the same general Rovy pattern as other Rovy-authored modules.

The transformer injects widget registration calls for each `@widget` builder class. Those registration side effects run when the module is required. `rovy.loadPaths(...)` is therefore the mechanism that auto-collects widgets:

```ts
rovy.loadPaths(
	"src/client/widgets",
	"src/client/screens",
);
```

No separate explicit widget manifest is planned.

## Plain-call authoring

The intended authored syntax is just a normal function call:

```ts
Window({ title: "Inventory" });
```

Not the primary surface:

```ts
RovyUi.Window({ title: "Inventory" });
```

Because widgets are plain functions, the authored call shape is already valid TypeScript. The transformer should recognize calls to JSDoc-tagged widget functions and lower them into the future widget-runtime invocation surface as needed.

The goal is that widget authoring feels like calling a widget with no `new`, no wrapper noise, and no typechecker panic.

## Caller-to-builder mapping

Caller-to-builder mapping is explicit and local.

Planned rule:

- `/** @widget WindowBuilder */` on `Window(...)` means "this caller maps to `WindowBuilder` in the same file"
- transformer resolves `WindowBuilder` with the TypeScript checker
- transformer must fail if the named builder is missing, not local to the same file, or not decorated as a widget builder

This avoids:

- class-call hacks that confuse TS
- cross-file string-path mapping
- inference from naming conventions alone

Same-file-only mapping is intentional. It keeps the rule simple and easy to validate.

## Statelessness rules

Widgets are stateless by public contract.

That means:

- no persistent widget-local app state fields
- no hidden hook-style storage
- no effect lifecycle API
- no build-time or runtime promise that widgets remember previous app behavior

Acceptable dynamic inputs:

- widget call args
- `Res<T>`, `ResMut<T>`, `OptRes<T>` injected into the builder `build(...)`
- `Query<...>` injected into the builder `build(...)`
- commands/events or explicit outbound actions
- external stores owned by other runtime systems

If the runtime eventually needs internal caches or identity helpers for mounting/reconciliation, those are implementation details only. They must not weaken the public stateless-widget contract.

## Compile-time keys

Compile-time keys are optional future UI metadata, not a hook system.

If kept, their role is limited to:

- widget identity for reconciliation
- stable transformer metadata
- diagnostics / debug labeling

They are explicitly **not** for:

- hook slots
- effect slots
- dependency arrays
- widget-local state buckets

## Planned transformer duties

UI adds a new planned transformer responsibility set:

1. Detect JSDoc `@widget BuilderName` on function declarations/exports.
2. Resolve the named builder in the same file only.
3. Validate that the named builder is an `@widget` class with `build(...)`.
4. Inject widget registry calls for widget builder classes.
5. Detect plain calls to tagged widget functions, such as `Window(args)`.
6. Lower those calls into the future widget-runtime invocation surface.
5. Optionally attach stable compile-time widget identity metadata.

Explicit non-goal:

- no hook/state/effect transform work for UI

## Planned validation and examples

This docs pass should drive later focused validation, not broad runtime work.

Planned future proof points:

- transformer fixture proving JSDoc caller-to-builder mapping
- transformer fixture proving `@widget` builder registration injection
- transformer fixture proving `Window(args)` lowers correctly
- example doc showing resource/arg-driven widget authoring without hooks
- regression note that widgets stay plain functions with no class-construction path

## Open questions

These remain open until implementation starts:

- exact widget runtime invocation function name / contract
- exact `@widget` registry metadata shape
- exact allowed injected params for `build(...)`
- how much widget identity metadata is required for reconciliation

Even with those open questions, the following are locked by this doc:

- widgets use a JSDoc caller function plus same-file builder class
- widget discovery is auto-registration via `rovy.loadPaths(...)`
- authoring uses plain function calls
- JSDoc `@widget BuilderName` is explicit mapping metadata
- builder lookup is same-file-only
- the builder class owns required `build(...)`
- the widget model is stateless
